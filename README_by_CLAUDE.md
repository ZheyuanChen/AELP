# Custom Laser Profile: Configurable Filename

## Summary of Code Modifications

Four source files were modified to allow the input deck to specify the data file used by the custom laser profile loaders, instead of relying on hardcoded filenames.

### `src/shared_data.F90`

Added a new `CHARACTER(LEN=c_max_path_length)` field `profile_data_file` to the `laser_block` derived type, alongside the existing `use_custom_profile` and `use_spatiotemporal` flags. Default value is blank (a single space), which signals "use the legacy default filename" at load time.

### `src/deck/deck_laser_block.f90`

- **`laser_block_start`**: Explicitly initialises `profile_data_file` to blank for each new laser block, consistent with the existing practice of initialising the other custom-profile flags here rather than relying on Fortran default initialisers.
- **`laser_block_handle_element`**: Added a new parser clause for the keyword `profile_data_file`. The value string is trimmed and left-adjusted before being stored on the working laser.

### `src/user_interaction/custom_laser.f90`

- **`custom_laser_spatial_setup` (1D path)**: The filename is now resolved from `laser%profile_data_file` if it is non-blank, otherwise falls back to the legacy default `spatial_profile.dat`. Absolute paths (starting with `/`) are used directly; relative paths are prepended with `data_dir`.
- **`load_temporal_spatial_profile` (2D path)**: Changed from a zero-argument subroutine to accepting a `CHARACTER(LEN=*)` filename argument. The same absolute/relative path resolution logic applies. The `profile_loaded` guard is unchanged — the first laser to trigger the load determines which file is read.
- **`custom_laser_profile`**: Now resolves the filename from `laser%profile_data_file` (defaulting to `temporal_spatial_profile.dat` if blank) and passes it to `load_temporal_spatial_profile`.

All changes are fully backward-compatible: omitting `profile_data_file` from the deck reproduces the original behaviour exactly.

## Usage Instructions

### New deck keyword

A new optional keyword `profile_data_file` is available inside `begin:laser` / `end:laser` blocks. It specifies the filename (or full path) of the custom profile data file to load.

**Syntax**

```
profile_data_file = <filename_or_path>
```

The value can be:

- **A plain filename** (e.g. `TS01.dat`): the file is expected in the same directory as the simulation output, i.e. the directory specified by `data_dir` / `USE_DATA_DIRECTORY`.
- **An absolute path** (e.g. `/home/user/profiles/TS01.dat`): the file is read from the specified location directly, regardless of `data_dir`.

If the keyword is omitted entirely, the original default filenames are used:

- `temporal_spatial_profile.dat` for the 2D spatiotemporal path
- `spatial_profile.dat` for the 1D spatial path

### Examples

**2D spatiotemporal profile with a custom filename:**

```
begin:laser
  boundary = x_min
  intensity_w_cm2 = 1.0e18
  lambda = 0.8e-6
  phase = ...
  use_custom_profile = T
  use_spatiotemporal_profile = T
  profile_data_file = TS01.dat
end:laser
```

This reads the 2D envelope from `<data_dir>/TS01.dat` instead of `<data_dir>/temporal_spatial_profile.dat`.

**1D spatial profile with a custom filename:**

```
begin:laser
  boundary = x_min
  intensity_w_cm2 = 1.0e18
  lambda = 0.8e-6
  use_custom_profile = T
  use_spatiotemporal_profile = F
  profile = custom
  profile_data_file = my_spatial_profile.dat
end:laser
```

**Using an absolute path:**

```
begin:laser
  boundary = x_min
  intensity_w_cm2 = 1.0e18
  lambda = 0.8e-6
  use_custom_profile = T
  use_spatiotemporal_profile = T
  profile_data_file = /shared/laser_profiles/focused_beam.dat
end:laser
```

**Backward-compatible (no change needed to existing decks):**

```
begin:laser
  boundary = x_min
  intensity_w_cm2 = 1.0e18
  lambda = 0.8e-6
  use_custom_profile = T
  use_spatiotemporal_profile = T
  # profile_data_file omitted — uses temporal_spatial_profile.dat
end:laser
```

### Data file format

The data file format is unchanged regardless of the filename.

The **2D spatiotemporal** file must follow:

```
Line 1:   N_t  N_y                          (integer counts)
Line 2:   y_0  y_1  ...  y_{N_y-1}          (spatial coordinates, metres)
Line 3:   t_0  t_1  ...  t_{N_t-1}          (time coordinates, seconds)
Lines 4+: N_t rows, each with N_y values    (normalised envelope in [0,1])
```

The **1D spatial** file must follow:

```
Line 1:   N_points                           (integer count)
Lines 2+: y_i  value_i                       (coordinate-value pairs)
```

### Notes

- The keyword is **per-laser**: each laser block can specify its own file.
- For the 2D spatiotemporal path, the profile data is loaded once into module-level storage (shared across all lasers). If multiple lasers specify different files, the first one to trigger the load wins.
- The file is read only by MPI rank 0 and broadcast to all ranks, consistent with the existing MPI discipline.
